# v3.0 — WXCODE Engine Integration

> Estender o wxcode-adm com campos, servicos, API e UI necessarios para que o
> wxcode engine opere em modo multi-tenant. O wxcode-adm ja tem auth, tenants,
> billing e super-admin. Este milestone adiciona a camada de integracao: tokens
> Claude per tenant, limites operacionais, provisioning e UI de gestao.

## Motivacao

O documento `wxcode/docs/MULTI-TENANT.md` define a arquitetura multi-tenant do
wxcode engine. O wxcode-adm (v1.0 + v2.0) ja implementa fundacao superior em
varios pontos (JWT RS256, N:N user-tenant, Stripe billing). Este milestone fecha
os gaps remanescentes para que o wxcode engine possa consumir dados do wxcode-adm.

Divergencias documentadas em: `wxcode/docs/MULTI-TENANT-ADJUSTMENTS.md`

## Phases

### Phase 20: Crypto Service + Tenant Model Extension
**Goal:** Adicionar campos Claude e wxcode ao model Tenant, com encriptacao Fernet para tokens sensiveis.

**Deliverables:**
- `backend/src/wxcode_adm/common/crypto.py` — Fernet encrypt/decrypt service
- `WXCODE_ENCRYPTION_KEY` em Settings (config.py)
- Novos campos no Tenant model:
  - `claude_oauth_token` (String, nullable, encrypted)
  - `claude_default_model` (String, default "sonnet")
  - `claude_max_concurrent_sessions` (Integer, default 3)
  - `claude_monthly_token_budget` (Integer, nullable = unlimited)
  - `database_name` (String, nullable, "wxcode_t_{slug}")
  - `default_target_stack` (String, default "fastapi-jinja2")
  - `neo4j_enabled` (Boolean, default True)
  - `status` (String: pending_setup | active | suspended | cancelled)
- Alembic migration 008
- Tests para crypto service
- Tests para novos campos

**Impacto:** Model + migration + service novo. Zero breaking changes nos endpoints existentes.

**Deliverables:**

- Tests

**Impacto:** Extensao do model Plan existente. Admin CRUD ja funciona, apenas recebe campos extras.

---

### Phase 22: Claude Provisioning API
**Goal:** Endpoints de super-admin para provisionar token Claude e configurar tenants para o wxcode engine.

**Deliverables:**
- `PUT /api/v1/admin/tenants/{id}/claude-token` — set/update encrypted token
- `DELETE /api/v1/admin/tenants/{id}/claude-token` — revoke token
- `PATCH /api/v1/admin/tenants/{id}/claude-config` — update model, sessions, budget
- `POST /api/v1/admin/tenants/{id}/activate` — transition pending_setup → active
- `GET /api/v1/tenants/{id}/wxcode-config` — config para wxcode engine (requer membership DEVELOPER+)
  - Retorna: database_name, default_target_stack, neo4j_enabled, claude_default_model, max_concurrent_sessions
  - NAO retorna: claude_oauth_token (token nunca sai do wxcode-adm, exceto em endpoint protegido engine-to-engine)
- Audit log entries para todas as operacoes de provisioning
- Tests

**Impacto:** Novos endpoints. Nao altera endpoints existentes.

### Phase 23: Admin UI — Claude Management
**Goal:** UI no super-admin portal para provisionar e gerenciar tokens Claude e limites wxcode.

**Deliverables:**
- Tenant detail page: secao "WXCODE Integration"
  - Form de Claude token (set/revoke, masked display)
  - Form de Claude config (model, sessions, budget)
  - Botao "Activate Tenant" (visivel quando status=pending_setup)
  - Badge de status (pending_setup=yellow, active=green, suspended=red)
- Plan form: campos de limites wxcode (max_projects, max_output_projects, max_storage_gb)
- Plan list: exibir limites wxcode nas colunas
- API hooks (TanStack Query) para novos endpoints
- Tests E2E basicos

**Impacto:** Extensao do frontend admin existente. Nao altera telas existentes.

### Phase 24: CORS Fix + Integration Contract
**Goal:** Corrigir CORS para producao e documentar contrato de integracao wxcode-adm ↔ wxcode engine.

**Deliverables:**
- Corrigir `main.py`: usar `settings.ALLOWED_ORIGINS` em vez de `["*"]` hardcoded
- Suporte a custom domains: `Tenant.wxcode_url` adicionado a CORS origins dinamicamente
- Health check endpoint para wxcode engine: `GET /api/v1/integration/health`
  - Retorna: versao wxcode-adm, status do servico, JWKS URL
- Documentar contrato de integracao (como wxcode engine consome wxcode-adm):
  - Autenticacao: JWT RS256 validado via JWKS
  - Tenant context: header X-Tenant-ID
  - Config: GET /tenants/{id}/wxcode-config
  - Token exchange: POST /auth/wxcode/exchange
- Tests

**Impacto:** Fix de seguranca (CORS). Novos endpoints de integracao.

## Dependencies

```
Phase 20 ──→ Phase 22 ──→ Phase 23
Phase 21 ──→ Phase 23
Phase 20 ──→ Phase 24
```

- Phase 20 e 21 podem ser executadas em paralelo
- Phase 22 depende de Phase 20 (crypto service + tenant fields)
- Phase 23 depende de Phase 22 (admin API) e Phase 21 (plan limits)
- Phase 24 pode ser executada apos Phase 20
- Phase 26 depende de Phase 23 (migration 010 que removeu token_quota)

## Success Criteria

1. Super-admin pode provisionar Claude token para um tenant via UI
2. Super-admin pode configurar limites wxcode por plan via UI
3. Super-admin pode ativar tenant (pending_setup → active) via UI
4. wxcode engine pode obter configuracao do tenant via API autenticada
5. CORS configurado para producao (nao wildcard)
6. Contrato de integracao documentado
7. Todos os tokens Claude encriptados em repouso (Fernet AES)
8. Audit log registra toda operacao de provisioning

## Out of Scope

- Provisioning automatico de MongoDB databases (wxcode engine faz isso)
- Execucao de conversoes (wxcode engine)
- Rate limiting per-tenant (excluido por decisao)
- API keys per tenant (carry-over de v1.0, milestone separado)
