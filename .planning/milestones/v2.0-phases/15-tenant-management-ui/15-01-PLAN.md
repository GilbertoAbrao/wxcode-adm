Create `frontend/src/hooks/useTenant.ts` following the exact pattern from `useUserAccount.ts` (useQuery for reads, useMutation with queryClient.invalidateQueries for writes).

**Critical: X-Tenant-ID header injection.** The backend tenant endpoints (except `/tenants/me`) require an `X-Tenant-ID` header. Since the project constraint is "usuario isolado a um tenant" (user belongs to exactly one tenant), the approach is:

1. `useMyTenants()` — `useQuery` calling `GET /tenants/me` (no X-Tenant-ID needed). Returns `{ tenants: MyTenantItem[] }`. This provides the tenant ID for all other hooks.

2. Create a helper function `tenantHeaders(tenantId: string)` that returns `{ headers: { "X-Tenant-ID": tenantId } }` to merge into apiClient options. The apiClient already merges caller-provided headers (see api-client.ts lines 49-53).

3. All tenant-scoped hooks accept `tenantId: string | undefined` as first parameter and use `enabled: !!tenantId` to prevent firing before tenant context is resolved.

**Hooks to create (matching backend router.py endpoints):**

Query hooks:
- `useMyTenants()` — GET /tenants/me, staleTime 60s, queryKey: ["tenants", "me"]
- `useTenantMembers(tenantId)` — GET /tenants/current/members, staleTime 15s, queryKey: ["tenant", tenantId, "members"]. Requires X-Tenant-ID header. enabled: !!tenantId
- `useTenantInvitations(tenantId)` — GET /tenants/current/invitations, staleTime 15s, queryKey: ["tenant", tenantId, "invitations"]. Requires X-Tenant-ID. enabled: !!tenantId

Mutation hooks:
- `useInviteMember(tenantId)` — POST /tenants/current/invitations, body: { email, role, billing_access }. On success: invalidate both ["tenant", tenantId, "members"] and ["tenant", tenantId, "invitations"]
- `useCancelInvitation(tenantId)` — DELETE /tenants/current/invitations/{id}. On success: invalidate ["tenant", tenantId, "invitations"]
- `useChangeRole(tenantId)` — PATCH /tenants/current/members/{user_id}/role, body: { role, billing_access }. On success: invalidate ["tenant", tenantId, "members"]
- `useRemoveMember(tenantId)` — DELETE /tenants/current/members/{user_id}. On success: invalidate ["tenant", tenantId, "members"]
- `useMfaEnforcement(tenantId)` — PATCH /tenants/current/mfa-enforcement, body: { enforce: boolean }. On success: invalidate ["tenants", "me"] (mfa_enforced is on tenant)

The `tenantHeaders` helper pattern:
```typescript
function tenantHeaders(tenantId: string): { headers: Record<string, string> } {
  return { headers: { "X-Tenant-ID": tenantId } };
}
```

Example hook using it:
```typescript
export function useTenantMembers(tenantId: string | undefined) {
  return useQuery<TenantMember[], Error>({
    queryKey: ["tenant", tenantId, "members"],
    queryFn: () =>
      apiClient<TenantMember[]>("/tenants/current/members", {
        ...tenantHeaders(tenantId!),
      }),
    enabled: !!tenantId,
    staleTime: 15_000,
  });
}
```

For mutation hooks that need tenantId, accept it as parameter:
```typescript
export function useInviteMember(tenantId: string | undefined) {
  const queryClient = useQueryClient();
  return useMutation<MembershipResponse, Error, InviteMemberRequest>({
    mutationFn: (data) =>
      apiClient<MembershipResponse>("/tenants/current/invitations", {
        method: "POST",
        body: JSON.stringify(data),
        ...tenantHeaders(tenantId!),
      }),
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ["tenant", tenantId, "members"] });
      queryClient.invalidateQueries({ queryKey: ["tenant", tenantId, "invitations"] });
    },
  });
}
```

For `useRemoveMember` and `useChangeRole`, the mutation variable includes the user_id to build the URL dynamically.
