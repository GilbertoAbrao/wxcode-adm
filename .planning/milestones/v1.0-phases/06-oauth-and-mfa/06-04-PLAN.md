       **enable_mfa_enforcement(db, redis, tenant, actor_user) -> None**:
       - Load the actor User from DB (db.get(User, actor_user.id) or use the user directly if passed)
       - Per locked decision: "Owner must have MFA enabled on their own account before they can turn on enforcement"
         - If not actor_user.mfa_enabled: raise MfaRequiredError() (imported from auth/exceptions.py — "MFA_REQUIRED", "Owner must enable MFA on their account first", 400)
       - Set tenant.mfa_enforced = True
       - Per locked decision: "Immediate lockout — active sessions are revoked, members without MFA must enroll on next login"
         - Query non-MFA user IDs: SELECT user_id FROM tenant_memberships JOIN users ON users.id = tenant_memberships.user_id WHERE tenant_memberships.tenant_id = tenant.id AND users.mfa_enabled = False
         - DELETE FROM refresh_tokens WHERE user_id IN (non_mfa_user_ids)
         - Also blacklist their active access tokens in Redis:
           For each non-MFA user_id, query their refresh tokens BEFORE deleting (to find associated access tokens). Actually, refresh tokens don't store access token JTIs. Instead, rely on the fact that deleted refresh tokens prevent token refresh, and access tokens expire in ACCESS_TOKEN_TTL_HOURS. This is acceptable — immediate lockout means they can't refresh, and short-lived access tokens expire naturally.
       - Import: `from sqlalchemy import delete, select` (already imported), `from wxcode_adm.auth.models import RefreshToken, User`

       - No session revocation needed (users keep their sessions)

       **get_enforcing_tenants_for_user(db, user_id) -> list[UUID]** — Helper for login flow:
       - Query: SELECT t.id FROM tenants t JOIN tenant_memberships tm ON t.id = tm.tenant_id WHERE tm.user_id = user_id AND t.mfa_enforced = True
       - Returns list of tenant UUIDs where MFA is enforced
       - Used by login flow to determine if MFA must be required even though user might not have personal MFA enabled

    1. **Modify login() in auth/service.py** — Add tenant enforcement check:

       After checking `user.mfa_enabled`, also check if any tenant the user belongs to enforces MFA:
       - Call `get_enforcing_tenants_for_user(db, user.id)` (lazy import from tenants.service to avoid circular import, same pattern as auto_join_pending_invitations)
       - If user has enforcing tenants AND user.mfa_enabled is False:
         - Return `mfa_required=True` with a special signal indicating `mfa_setup_required=True` — the user needs to enroll in MFA before they can access enforcing tenants
         - Store in Redis: `auth:mfa_pending:{token}` with value JSON `{"user_id": str(user.id), "setup_required": true}` and TTL 300s
         - The client knows to redirect to MFA enrollment instead of TOTP entry
       - If user has enforcing tenants AND user.mfa_enabled is True:
         - Per locked decision: "No remember-device when tenant enforces MFA"
         - Skip the trusted device check when any enforcing tenant exists
         - Always require TOTP

       IMPORTANT: The enforcement check must happen AFTER password verification (don't leak info about enforcement to invalid credentials). The flow is:
       1. Verify credentials (existing)
       2. Check email verified (existing)
       3. Check if user is in any enforcing tenant -> if yes AND no MFA -> return mfa_setup_required
       4. Check if user has MFA enabled -> if yes:
          a. If in enforcing tenant -> skip device trust, always require TOTP
          b. If NOT in enforcing tenant -> check device trust, skip if trusted
          c. Otherwise -> require TOTP
       5. If no MFA and no enforcement -> issue tokens normally

    2. **Modify mfa_verify() in auth/service.py** — Handle setup_required flow:

       When the Redis mfa_pending value contains `"setup_required": true`, the verify function should reject the attempt and return an error indicating the user must complete MFA enrollment first (via POST /auth/mfa/enroll + POST /auth/mfa/confirm). After enrollment, the user must log in again.

       Actually, simpler: when login detects enforcement + no MFA, it can issue a temporary "mfa_setup" token. The client flow is:
       1. Login -> gets mfa_setup_required signal
       2. Client uses existing access token (temporary, limited scope) to call /auth/mfa/enroll and /auth/mfa/confirm
       3. After enrollment confirmed, client calls login again -> now user has MFA -> normal MFA flow

       But wait, the user doesn't have an access token yet (login didn't issue one). Instead:
       - Store the mfa_pending token in Redis as before
       - When mfa_verify is called with this token, check if the Redis value has setup_required=true
       - If setup_required, raise a specific error telling the user to enroll first
       - After user enrolls (using a separate auth mechanism — they can use the mfa_pending token as a short-lived bearer), they call login again

       Simplest approach: Return LoginResponse with `mfa_required=True, mfa_token=token, mfa_setup_required=True`. The frontend knows to redirect to enrollment. After enrollment, the user logs in again. The mfa_pending token from the first attempt expires naturally (300s).

       Add `mfa_setup_required: bool = False` field to LoginResponse schema (in auth/schemas.py).

    3. **Modify resolve_oauth_account() in auth/service.py** — OAuth onboarding adjustments:

       Per locked decision: "Same onboarding flow as email/password users, but skip workspace creation if already invited to a tenant"
       - After creating a new OAuth user, check if they have pending invitations (query Invitation by email where accepted_at IS NULL)
       - If pending invitations exist, set `needs_onboarding=False` in the response (frontend skips workspace creation)
       - If no pending invitations, set `needs_onboarding=True` (frontend shows workspace creation)
       - The actual invitation acceptance happens during email OTP verification (auto_join_pending_invitations is already called by verify_email)

       Per locked decision: "Still require email OTP verification" — this is already handled in Plan 01 (OAuth creates unverified user + sends OTP). No change needed here.

       Per locked decision: "If OAuth provider account is deleted/suspended and no password set, user can use the password reset flow to set a password"
       - The existing forgot_password function uses `user.password_hash` as salt for itsdangerous
       - For OAuth-only users with password_hash=None, use a fixed salt like `"no-password-{user_id}"` instead
       - Modify `generate_reset_token` and `verify_reset_token` to handle None password_hash:
         ```python
         def _reset_salt(user):
             return user.password_hash if user.password_hash else f"no-password-{user.id}"
         ```
       - This allows OAuth-only users to receive a reset link and SET a password (which then invalidates the token via the new hash as salt)
