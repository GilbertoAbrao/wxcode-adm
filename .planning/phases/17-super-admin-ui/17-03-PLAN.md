  <name>Task 1: Create useAdminUsers hooks for user search, detail, and moderation</name>
  <files>frontend/src/hooks/useAdminUsers.ts</files>
  <action>
    Create TanStack Query hooks for admin user management endpoints, following the same pattern as useAdminTenants.ts.

    **TypeScript interfaces** (matching backend admin/schemas.py exactly):
    - `UserListItem`: { id: string, email: string, display_name: string | null, email_verified: boolean, is_active: boolean, mfa_enabled: boolean, created_at: string }
    - `UserListResponse`: { items: UserListItem[], total: number }
    - `UserMembershipItem`: { tenant_id: string, tenant_name: string, tenant_slug: string, role: string, billing_access: boolean, is_blocked: boolean }
    - `UserSessionItem`: { id: string, device_type: string | null, browser_name: string | null, ip_address: string | null, city: string | null, last_active_at: string | null }
    - `UserDetailResponse`: { id: string, email: string, display_name: string | null, avatar_url: string | null, email_verified: boolean, is_active: boolean, is_superuser: boolean, mfa_enabled: boolean, created_at: string, updated_at: string, memberships: UserMembershipItem[], sessions: UserSessionItem[] }
    - `UserBlockRequest`: { tenant_id: string, reason: string }
    - `UserUnblockRequest`: { tenant_id: string, reason: string }

    **Query key constants:**
    ```
    ADMIN_USER_KEYS = {
      list: (params) => ["admin", "users", params] as const,
      detail: (userId: string) => ["admin", "users", userId] as const,
    }
    ```

    **Hooks:**
    - `useAdminUsers(params: { limit?: number, offset?: number, q?: string | null })` — useQuery calling GET /admin/users with query params (q for search). Build URL with URLSearchParams, skip null/undefined/empty params. staleTime: 30s. Returns UserListResponse. The query should be enabled even when q is empty/null (shows all users).
    - `useAdminUserDetail(userId: string | null)` — useQuery calling GET /admin/users/{userId}. enabled: !!userId. staleTime: 30s. Returns UserDetailResponse.
    - `useBlockUser()` — useMutation calling POST /admin/users/{user_id}/block with { tenant_id, reason } body. Variables type: `{ user_id: string, tenant_id: string, reason: string }`. On success: invalidate `["admin", "users"]` queries (both list and detail).
    - `useUnblockUser()` — useMutation calling POST /admin/users/{user_id}/unblock with { tenant_id, reason } body. Same pattern as useBlockUser.

    All hooks use `adminApiClient` from `@/lib/admin-api-client`.
    "use client" directive at top.
  </action>

    Run `cd frontend && npx tsc --noEmit` — zero TypeScript errors. Verify useAdminUsers.ts exports the 4 hooks and all TypeScript interfaces.
  </verify>

    Create the /admin/users page with email search, user list, detail drawer, and moderation actions.

    **Page structure:**
    - "use client" directive
    - Page header: "User Management" title with description
    - Same admin navigation bar as tenants page: horizontal nav with /admin/tenants and /admin/users links (users active here, cyan-400), Logout button on right. Copy the nav pattern from tenants page — or better, extract as inline component since both pages use it.

    **Search bar:**
    - GlowInput with Search icon (from lucide-react), placeholder "Search by email..."
    - Debounced search: use a local state for the input value and a separate `debouncedQuery` state updated after 300ms delay (useEffect with setTimeout). Pass debouncedQuery to useAdminUsers({ q: debouncedQuery }).
    - Clear button (X icon) to reset search

    **User table:**
    - Responsive table (overflow-x-auto)
    - Columns: Email, Display Name, Verified, MFA, Created, (clickable row)
    - Email column: primary text, clickable (sets selectedUserId for drawer)
    - Display Name: text or "—" if null
    - Verified: green checkmark or red X icon (CheckCircle2 / XCircle from lucide)
    - MFA: Shield icon green if enabled, zinc if disabled
    - Created: formatted date
    - Row click: sets `selectedUserId` state → opens detail drawer
    - Row hover: bg-zinc-900/30 cursor-pointer

    **Table styling:** Same dark theme as tenants page (bg-zinc-900/50 header, border-zinc-800 rows)

    **Pagination:** Same pattern as tenants page — page state, limit=20, Previous/Next buttons, "Showing X-Y of Z users"

    **User detail drawer:**
    - State: `selectedUserId: string | null`
    - When selectedUserId is set, show a slide-out panel from the right side
    - Drawer implementation: fixed right-0 top-0 h-full w-full sm:w-96 bg-zinc-950 border-l border-zinc-800 z-50, with a semi-transparent backdrop overlay (fixed inset-0 bg-black/50) that closes drawer on click
    - Transition: use CSS transition (translate-x-full when hidden, translate-x-0 when visible) with transition-transform duration-300
    - Close button (X icon) at top-right of drawer
    - Content from useAdminUserDetail(selectedUserId):
      - **User header:** Avatar (if avatar_url, else initials circle), email, display_name, badges for verified/MFA/superuser
      - **Account info section:** ID (truncated UUID), created_at, updated_at, is_active badge
      - **Memberships section:** List of UserMembershipItem entries:
        - Each row: tenant_name (tenant_slug), role badge (same roleBadgeClass colors as team page: owner=cyan, admin=purple, other=zinc), billing_access indicator
        - Blocked status: if is_blocked, show red "Blocked" badge
        - Action buttons per membership:
          - If NOT blocked: "Block" button (rose text)
          - If blocked: "Unblock" button (emerald text)
        - On block/unblock click: show inline reason input (same pattern as tenants suspend/reactivate) — state: `blockAction: { user_id: string, tenant_id: string, action: "block" | "unblock" } | null`
        - Confirm: call useBlockUser or useUnblockUser with { user_id, tenant_id, reason }
        - On success: clear blockAction state, mutations invalidate queries so drawer refreshes
      - **Sessions section:** List of UserSessionItem entries:
        - Each row: device_type, browser_name, ip_address, city, last_active_at (formatted relative or absolute date)
        - Read-only (admin can view but not revoke user sessions from this UI — not in requirements)

    **Loading/Error/Empty states:**
    - User list: LoadingSkeleton / ErrorState / EmptyState ("No users found")
    - Drawer detail: LoadingSkeleton inside drawer while loading, ErrorState on error
