<objective>
Create the admin section foundation: isolated admin auth (separate from tenant user auth), admin API client, admin login page at /admin/login, and admin route protection via AdminAuthProvider.

Purpose: The admin portal must be completely isolated from the tenant-facing app — separate token storage, separate API client, separate route protection, separate layout. This ensures admin-audience JWTs never mix with user-audience JWTs and the two auth contexts are independent.

Output: Admin login page at /admin/login that authenticates against POST /api/v1/admin/login and stores admin-audience JWTs in a separate in-memory store; unauthenticated access to /admin/* redirects to /admin/login.
</objective>

  <action>
    Create three files following the exact patterns from the tenant-user auth system (auth.ts, api-client.ts, useAuth.ts) but completely isolated for admin:

    **admin-auth.ts** — In-memory admin token storage, mirroring auth.ts but with separate module-scoped variables:
    - `_adminAccessToken` and `_adminRefreshToken` (module-scoped, NOT shared with user tokens)
    - Export: `getAdminAccessToken()`, `getAdminRefreshToken()`, `setAdminTokens(access, refresh)`, `clearAdminTokens()`, `isAdminAuthenticated()`
    - Export: `refreshAdminTokens()` — calls POST /api/v1/admin/refresh with the admin refresh token; updates stored tokens on success; returns boolean
    - Use the same API_BASE pattern: `process.env.NEXT_PUBLIC_API_URL || "http://localhost:8040"`
    - CRITICAL: These are completely separate variables from the user auth store. Admin login does NOT affect user login state and vice versa.

    **admin-api-client.ts** — Typed fetch wrapper for admin endpoints, mirroring api-client.ts:
    - Import from `admin-auth.ts` (NOT from auth.ts)
    - Export: `adminApiClient<T>(endpoint, options)` — works like apiClient but:
      - Prepends `/api/v1` to endpoints (same as apiClient)
      - Injects Authorization header from `getAdminAccessToken()` (not getAccessToken)
      - On 401: attempts `refreshAdminTokens()`, retries once, then calls `clearAdminTokens()`
      - skipAuth option for the login mutation
    - Reuse the same `ApiError` class from `@/lib/api-client` (import it, do not duplicate)
    - Import parseErrorBody logic — since it's not exported from api-client.ts, implement a local copy of the same parseErrorBody helper

    **useAdminAuth.ts** — TanStack Query mutations for admin auth endpoints:
    - TypeScript interfaces matching backend admin schemas:
      - `AdminLoginRequest`: { email: string, password: string }
      - `AdminTokenResponse`: { access_token: string, refresh_token: string, token_type: string }
    - Export `useAdminLogin()` — calls POST /admin/login via adminApiClient with skipAuth: true
    - Export `useAdminLogout()` — calls POST /admin/logout via adminApiClient (requires auth)
    - Follow exact same pattern as useAuth.ts mutations
  </action>
