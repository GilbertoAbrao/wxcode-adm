         f. **30-day trend (Python-side grouping — NOT PostgreSQL date_trunc per research anti-patterns):**
            - Load ALL subscriptions (active + canceled) with created_at in last 30 days or still active.

Note: TenantSubscription does not have a `canceled_at` field — use `updated_at` when status == CANCELED as a proxy, OR simply iterate subs and check status/period. Simpler approach: just compute current snapshot + the most recent change date. Actually, the research says "calculate from TenantSubscription history (created_at, canceled_at timestamps)". Since there is no canceled_at column, use a simpler trend: query `TenantSubscription` and compute cumulative totals per day from created_at. For each day d in [now-30..now], count subs where `created_at <= d` and `(status != CANCELED or updated_at > d)`. Use Python datetime iteration. If this is too slow for large datasets, it's fine for the dashboard — we're querying a small number of subscriptions.
            - Build list of `MRRTrendPoint(date=d.isoformat(), mrr_cents=..., active_count=...)`.
