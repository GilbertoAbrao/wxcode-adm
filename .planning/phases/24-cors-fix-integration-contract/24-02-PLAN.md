**1. Add integration health endpoint to common/router.py:**

Add a new endpoint `GET /integration/health` to the existing `router` in `common/router.py`. This endpoint is for the wxcode engine to discover wxcode-adm capabilities.

```python
@router.get("/integration/health")
@limiter.exempt
async def integration_health(
    db: AsyncSession = Depends(get_session),
    redis: Redis = Depends(get_redis),
) -> dict:
    """
    Integration health endpoint for wxcode engine.

    Returns service status, version, and JWKS URL for JWT validation.
    No authentication required — this is a discovery endpoint.
    """
```

Response format:
```json
{
    "service": "wxcode-adm",
    "version": "0.1.0",
    "status": "healthy",
    "jwks_url": "/.well-known/jwks.json",
    "endpoints": {
        "wxcode_config": "/api/v1/tenants/{tenant_id}/wxcode-config",
        "token_exchange": "/api/v1/auth/wxcode/exchange",
        "health": "/api/v1/health"
    }
}
```

Implementation:
- Version comes from the FastAPI app or a constant. Use `"0.1.0"` directly (matching main.py `version="0.1.0"`).
- Status: perform the same PostgreSQL + Redis checks as the existing `/health` endpoint. Return `"healthy"` if both pass, `"degraded"` if Redis fails but PostgreSQL works, `"unhealthy"` if PostgreSQL fails.
- `jwks_url`: hardcoded `"/.well-known/jwks.json"` (it's a well-known path, not configurable).
- `endpoints`: hardcoded dictionary of key integration endpoint paths. This acts as a discovery document.
- Rate limit exempt (same as /health — monitoring/discovery endpoints should not be throttled).

  <action>
Create `docs/INTEGRATION-CONTRACT.md` documenting how the wxcode engine consumes wxcode-adm.

The document should cover the following sections:

**1. Overview**
- wxcode-adm is the authentication and tenant management layer
- wxcode engine consumes wxcode-adm for: JWT validation, tenant configuration, token exchange
- wxcode-adm never executes wxcode engine operations

**2. Authentication: JWT RS256 via JWKS**
- wxcode engine validates JWTs locally using the public key from `GET /.well-known/jwks.json`
- Token format: RS256, contains `sub` (user_id), `tenant_id`, `role`, `aud` ("wxcode-adm"), `exp`, `jti`
- Key rotation: `kid` header in JWT matches a key in the JWKS response
- No token introspection — fully self-contained validation

**3. Tenant Context**
- All tenant-scoped requests include `X-Tenant-ID` header (UUID)
- wxcode engine extracts tenant_id from JWT claims and/or header
- Tenant isolation: each tenant has its own database (Tenant.database_name)

**4. Configuration Endpoint**
- `GET /api/v1/tenants/{tenant_id}/wxcode-config`
- Requires: valid JWT with DEVELOPER+ role, X-Tenant-ID header matching path
- Returns: database_name, default_target_stack, neo4j_enabled, claude_default_model, max_concurrent_sessions
- Does NOT return: claude_oauth_token (never leaves wxcode-adm)

**5. Token Exchange**
- `POST /api/v1/auth/wxcode/exchange`
- Request body: `{ "code": "<one-time-code>" }`
- Response: `{ "access_token": "...", "refresh_token": "..." }`
- Flow: User logs in at wxcode-adm -> redirected to wxcode with ?code= -> wxcode backend exchanges code for tokens
- Code is single-use (atomically consumed via Redis GETDEL)
- TTL: 30 seconds (configurable via WXCODE_CODE_TTL)

**6. Discovery**
- `GET /api/v1/integration/health` — service status, version, JWKS URL, endpoint map
- `GET /api/v1/health` — infrastructure health (PostgreSQL, Redis)

**7. Error Format**
- All errors return: `{ "error_code": "SNAKE_CASE_CODE", "message": "Human-readable description" }`
- Common codes: TENANT_NOT_FOUND, FORBIDDEN, INVALID_CODE, TOKEN_EXPIRED

**8. Rate Limits**
- Auth endpoints: 5/min per IP
- Global default: 60/min per IP
- Health/integration endpoints: exempt

Keep the document concise and developer-focused. Use code blocks for request/response examples. This is a reference document for the wxcode engine team.
