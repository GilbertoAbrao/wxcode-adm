2. **auth/service.py** — Add forgot_password and reset_password functions:

   **Module-level serializer:**
   - `from itsdangerous import URLSafeTimedSerializer`
   - `from itsdangerous.exc import SignatureExpired, BadSignature`
   - Create serializer: `reset_serializer = URLSafeTimedSerializer(settings.JWT_PRIVATE_KEY.get_secret_value(), salt="password-reset")`
     - Using JWT_PRIVATE_KEY as the secret is fine — it is a strong random secret that already exists in settings. The `salt="password-reset"` namespace separates it from JWT usage.

   **Token generation helper:**
   - `def generate_reset_token(email: str, pw_hash: str) -> str`:
     - `return reset_serializer.dumps(email, salt=pw_hash)`
     - The user's current password_hash is used as the salt — changing the password automatically invalidates the token (single-use enforcement)

   **Token verification helper:**
   - `def verify_reset_token(token: str, pw_hash: str) -> str`:
     - `return reset_serializer.loads(token, salt=pw_hash, max_age=86400)` — 24 hours = 86400 seconds
     - On `SignatureExpired`: raise `TokenExpiredError()`
     - On `BadSignature`: raise `InvalidTokenError()` — this catches both tampered tokens AND single-use (pw changed)
     - Returns the email string on success

   **Forgot password function:**
   - `async def forgot_password(db: AsyncSession, redis: Redis, body: ForgotPasswordRequest) -> None`:
     - Find user by email — if NOT found, return silently (enumeration prevention)
     - If found: generate reset token: `token = generate_reset_token(body.email, user.password_hash)`
     - Build reset link: for now, a placeholder URL pattern: `f"{settings.ALLOWED_ORIGINS[0]}/reset-password?token={token}"`
       - The actual frontend URL will be adjusted in Phase 7. For now, logging the token in the arq job is the primary mechanism.
     - Enqueue arq job: `pool = await get_arq_pool(); await pool.enqueue_job("send_reset_email", str(user.id), user.email, reset_link); await pool.aclose()`

   **Reset password function:**
   - `async def reset_password(db: AsyncSession, body: ResetPasswordRequest) -> None`:
     - We need to find the user FIRST to get their password_hash for token verification.
     - Strategy: The token contains the email (signed by itsdangerous). We can extract it WITHOUT verification first to look up the user, then verify with their current pw_hash.
     - Step 1: Extract email from token without full verification:
       ```python
       try:
           # loads without salt verification to get the email
           # We need the email to find the user and their pw_hash
           # Using a two-step approach: first unsign to get email, then verify with pw_hash
           email = reset_serializer.loads(token, salt="password-reset", max_age=86400)
           # Wait — this won't work because the token was signed with pw_hash as salt, not "password-reset"
       ```
     - CORRECT approach: The token was dumped with `salt=pw_hash`. We can't load it without the correct salt. But we need the email to get the pw_hash. Circular dependency.
     - SOLUTION per research: Use the `URLSafeTimedSerializer` differently. The `salt` param in `dumps()` affects the HMAC. But `dumps` encodes the payload (email) in the URL-safe token. We can unsign the outer layer to get the payload without verifying the inner salt.
     - Actually, the standard pattern is simpler: `serializer.loads_unsafe(token)` returns `(is_valid, payload)` regardless of signature.
     - Better: Use `loads_unsafe` to extract the email, find the user, then verify properly:
       ```python
       _, data = reset_serializer.loads_unsafe(token)
       if data is None:
           raise InvalidTokenError()
       email = data  # The payload is the email string
       ```
     - Step 2: Find user by email — if not found, raise `InvalidTokenError()`
     - Step 3: Verify token properly: `verify_reset_token(body.token, user.password_hash)`
       - This validates signature, expiry, AND single-use (if password already changed, pw_hash differs -> BadSignature)
     - Step 4: Update password: `user.password_hash = hash_password(body.new_password)`
     - Step 5: Revoke all sessions: `await db.execute(delete(RefreshToken).where(RefreshToken.user_id == user.id))`
     - Note: Access token blacklisting is not done here because we don't have the user's current access token. The 24h TTL will expire it naturally. If this is a security concern, Plan 05's `require_verified` dependency will add an is_active check.
