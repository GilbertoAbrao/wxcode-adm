**Section 1: Rate Limiting Tests (PLAT-03)** — Success Criteria 3

These tests need the limiter enabled. Use a fixture or context manager that temporarily sets `limiter.enabled = True` and uses an in-memory storage (no Redis needed). The simplest approach: create a fresh `Limiter` with `storage_uri="memory://"` and `default_limits=["2/minute"]` (low threshold for fast tests), swap it onto `app.state.limiter`, then restore after.

Tests:
- `test_auth_endpoint_rate_limited`: Enable limiter with strict limit (e.g., 2/minute for test speed). Hit POST /api/v1/auth/login repeatedly. Assert first 2 return 401 (invalid creds, but not 429). Assert 3rd returns 429 with Retry-After header.
- `test_health_endpoint_exempt`: Enable limiter with strict limit (2/minute). Hit GET /api/v1/health 5 times. All return 200 or 503 (health status) — never 429.
- `test_jwks_endpoint_exempt`: Enable limiter with strict limit. Hit GET /.well-known/jwks.json 5 times. All return 200 — never 429.
- `test_rate_limit_response_includes_retry_after`: Verify the 429 response includes the `Retry-After` header.

**Section 2: Audit Log Tests (PLAT-04)** — Success Criteria 4

- `test_write_audit_creates_entry`: Call `write_audit(db, actor_id=user.id, action="login", resource_type="user", resource_id=str(user.id), tenant_id=None, ip_address="127.0.0.1", details={"method": "password"})`. Commit. Query audit_logs table. Assert 1 row with correct fields.
- `test_write_audit_system_action_no_actor`: Call write_audit with actor_id=None (system action). Assert entry is created with actor_id NULL.
- `test_write_audit_details_stored_as_json`: Write an entry with details={"role": "admin", "invited_by": "owner"}. Read back. Assert details dict matches.
- `test_purge_old_audit_logs`: Create 3 audit entries. Manually backdate 2 of them by setting created_at to > AUDIT_LOG_RETENTION_DAYS ago. Call purge_old_audit_logs with a mock ctx containing session_maker. Assert only 1 entry remains.
- `test_audit_log_query_superadmin`: Create a super-admin user, get auth token. Create some audit entries. GET /api/v1/admin/audit-logs. Assert 200 with correct items.
- `test_audit_log_query_non_superadmin_forbidden`: Create a regular user, get auth token. GET /api/v1/admin/audit-logs. Assert 403.
- `test_audit_log_query_filtering`: Create entries with different actions. GET /api/v1/admin/audit-logs?action=login. Assert only matching entries returned.

**Section 3: Email Template Tests (PLAT-05)** — Success Criteria 5

Since email sending requires SMTP (which is not available in tests), test the template rendering and the sender function contracts:

- `test_verification_email_uses_html_template`: Patch `fast_mail.send_message` to capture the call. Call `send_verification_email(ctx={}, user_id="test", email="a@b.com", code="123456")`. Assert send_message was called with `html_template="verify_email.html"` and `plain_template="verify_email.txt"` kwargs. Assert `template_body` includes `{"code": "123456"}`.
- `test_reset_email_uses_html_template`: Same pattern for `send_reset_email`. Assert template_body includes `{"reset_link": "..."}`.
- `test_invitation_email_uses_html_template`: Same for `send_invitation_email`. Assert template_body includes `{"tenant_name": ..., "role": ..., "invite_link": ...}`.
- `test_payment_failed_email_uses_html_template`: Same for `send_payment_failed_email`. Assert template_body includes `{"tenant_name": ...}`.
- `test_all_html_templates_extend_base`: For each of the 4 HTML templates, read the file content and assert it contains `{% extends "base.html" %}`.
- `test_templates_use_direct_variables_not_body_prefix`: For each HTML template, read content and assert `{{ body.` does NOT appear anywhere (fastapi-mail 1.6.2 uses direct variables).

Use `unittest.mock.patch` or `monkeypatch` to mock `fast_mail.send_message`. The email functions catch all exceptions, so the mock must NOT raise. Import the sender functions from their respective modules (auth.email, tenants.email, billing.email).
